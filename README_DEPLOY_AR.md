# نشر هود كوم على Netlify

## الرفع الصحيح
1. فك ضغط الحزمة.
2. ارفع **محتويات مجلد `hud-com-netlify` كما هي مع المجلدات الداخلية** إلى Netlify، أو ارفع المجلد من Netlify Drop.
3. لا تجمع الملفات في مجلد واحد؛ مجلدات `pages` و`services` و`stores` و`utils` و`components` و`config` ضرورية.
4. إعداد النشر هو الجذر الحالي، وملف `netlify.toml` يحدد `publish = "."`.

## فحص قبل الرفع
```bash
npm run check
```
يجب أن تظهر رسالة نجاح فحص 50 وحدة JavaScript.

## اعتماد Supabase
Netlify يستضيف الواجهة فقط. التسجيل والدخول والأقسام والتعليقات ولوحة الإدارة تعتمد على مشروع Supabase المحدد في `config/supabase.js`، ويجب أن يكون:
- مشروع Supabase نشطًا وغير متوقف.
- جدول `hud_docs` موجودًا.
- سياسات RLS تسمح بالقراءة العامة للبيانات العامة، وتسمح للمستخدم الموثق بالعمليات الخاصة به، وتمنع صلاحيات الإدارة عن غير المدير.
- Site URL وRedirect URLs في Supabase Auth تحتوي رابط Netlify النهائي (مثل `https://your-site.netlify.app/**`).
- مزود البريد مفعّل إذا كان تأكيد البريد مطلوبًا.
- وثيقة بوابة الإدارة `settings/admin_gate` مهيأة في قاعدة البيانات؛ لا توجد بيانات إدارة ثابتة مكشوفة داخل ملفات الواجهة.

إذا كانت قاعدة Supabase أو سياسات RLS غير مهيأة فلن تستطيع أي ملفات واجهة إصلاح ذلك وحدها، وستظهر أخطاء API واضحة في Console بدل بقاء الواجهة معلقة.

## إصلاحات هذه الحزمة
- استعادة هيكل ES Modules الذي فُقد عند تسطيح الملفات، وهو السبب المباشر لفشل `main.js` على Netlify.
- إضافة ملف الصفحة الرئيسية المفقود `pages/index.js`.
- منع ظهور نموذج تعليق للزائر: الحاوية تبدأ بحالة تحقق ثم تُستبدل ببوابة تسجيل دخول أو نموذج المستخدم الموثق.
- التحقق مرة ثانية داخل خدمة إرسال التعليق، فلا يكفي تعديل DOM لتجاوز الحماية.
- إصلاح مسار فحص المشروع وإضافة إعداد Netlify ورؤوس MIME والحماية.
- إضافة `logo.svg` المفقود.
- حالات بديلة واضحة عند فشل تحميل الأقسام أو التعليقات بدل مؤشر تحميل دائم.
